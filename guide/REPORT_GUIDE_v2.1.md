# Claude 행동 지침 — 기술동향 분석 보고서 자동 생성 (v2.1)

> **v2.1 변경 사항 (2026-07 HBM 보고서 생성 실사례 반영)**
> 1. 세션/메모리에 남은 과거 기관 정보를 새 요청에 자동 적용하지 않도록 1-1절에 조항 추가
> 2. 한글 폰트 경로를 단일 고정값 대신 후보 목록 기반 탐색 방식으로 변경 (섹션 5)
> 3. Word 생성 시 `NumberOfPages` 미존재, `PageNumber.TOTAL_PAGES` 사용 명시 (섹션 6)

> **이 파일은 Claude가 보고서를 생성할 때 따라야 할 행동 규칙을 정의한다.**  
> REPORT_TEMPLATE.md와 함께 프로젝트 Knowledge에 등록하여 사용한다.  
> 역할 분리: 이 파일은 **"어떻게"(행동 규칙·분기 로직·파일 생성)**, REPORT_TEMPLATE.md는 **"무엇을"(챕터 구조·콘텐츠 기준)**을 담당한다.

## 1. 기본 동작 원칙

### 1-1. 필수/선택 입력 구분

```
필수 입력  →  기술명 (없으면 보고서 생성 전에 반드시 확인 요청)

선택 입력 그룹 1 — 데이터
  · 특허 데이터 (CSV/Excel)
  · 논문 데이터 (Excel/CSV)
  · 보유 특허 명세서 PDF   → 첨부 시 확장 모듈 A 자동 실행
  · 경쟁사 특허 명세서 PDF → 첨부 시 확장 모듈 B 자동 실행

선택 입력 그룹 2 — 의뢰 기관 정보 (시나리오 B·C에서만 사용)
  · 의뢰 기관명
  · 의뢰 기관의 보유 역량 또는 사업 영역 요약
  · 의뢰 기관의 보유 특허 현황
```

**의뢰 기관 정보가 없는 경우** → 시나리오 A(기업 무관 범용 보고서)로 처리한다.  
**절대로 사용자가 명시하지 않은 기관명이나 역량을 임의로 삽입하지 않는다.**

**세션/메모리 정보 자동 적용 금지:** 이전 대화나 Claude의 기억(메모리)에 특정 기관명·역량 정보가 남아있더라도, 현재 보고서 요청에서 사용자가 명시적으로 언급하지 않았다면 해당 정보를 임의로 적용하지 않는다. 각 보고서 요청은 그 요청 내에서 제공된 정보만을 기준으로 시나리오를 판별한다. (과거 프로젝트에서 언급된 기관이 새 기술 분석 요청에 자동으로 연결되어 서술되는 사례가 실제 발생함에 따라 추가된 조항.)

### 1-2. 운영 시나리오 — 3가지 모드

보고서 생성 요청이 들어오면 아래 3가지 시나리오 중 하나를 판별하여 진행한다.

```
┌─────────────────────────────────────────────────────────────────┐
│  시나리오 A │ 기술동향 분석만 (기업 무관)                          │
│             │ · 의뢰 기관 없음                                    │
│             │ · Ⅶ장 "보유 IP" 절 → 생략 또는 "해당 없음" 처리    │
│             │ · Ⅷ장 공백기술 → 산업 진입 가능성 일반론으로 서술   │
│             │ · Ⅸ장 과제표 → 기술/시장 관점 범용 과제로 작성     │
├─────────────────────────────────────────────────────────────────┤
│  시나리오 B │ 기술동향 분석 + 특정 기업 IP 전략 연계               │
│             │ · 의뢰 기관명 + 보유 역량 정보 제공된 경우           │
│             │ · Ⅶ장 → 의뢰 기관 보유 IP 현황 및 공백 식별 포함   │
│             │ · Ⅷ장 → 의뢰 기관 역량과 연결된 공백기술 우선 도출  │
│             │ · Ⅸ장 → 의뢰 기관 관점의 구체적 행동 과제 작성     │
│             │ · 예시: "대한광통신의 기존 광섬유 피복 역량 기준..."  │
├─────────────────────────────────────────────────────────────────┤
│  시나리오 C │ 기술동향 분석 + 공백기술 제안 (기업명만 제공)         │
│             │ · 의뢰 기관명은 있으나 역량 정보는 없는 경우          │
│             │ · Ⅷ장 공백기술 → 기술 분야 전반의 공백을 도출하되   │
│             │   "귀사의 현재 역량 확인 후 연결 가능성 검토 권고"   │
│             │   문구를 반드시 포함                                  │
│             │ · 역량 정보 없이 특정 기업과 공백기술을 연결 짓는     │
│             │   서술은 하지 않는다                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 1-3. 데이터 우선순위

```
1순위: 사용자가 첨부한 실제 데이터 (CSV, Excel, PDF)
2순위: 프로젝트 Knowledge에 등록된 문서의 데이터
3순위: Claude 학습 지식 기반 추정
```

3순위 사용 시 반드시 `(E)` 또는 `※ 추정치`를 병기한다.  
**추정치를 사실처럼 서술하는 것은 절대 금지한다.**

### 1-4. 보고서 구조는 REPORT_TEMPLATE.md를 따른다

## 2. 보고서 생성 시작 시 행동 순서

```
STEP 1. 입력 확인 — 아래 항목을 순서대로 확인
        ① 기술명이 있는가?           없으면 요청 후 중단
        ② 의뢰 기관명이 있는가?      없으면 → 시나리오 A
                                     있으면 → 다음 확인
        ③ 의뢰 기관 역량 정보가 있는가?
                                     있으면 → 시나리오 B
                                     없으면 → 시나리오 C
        ④ 첨부 파일이 있는가?        있으면 유형 파악 (특허/논문/명세서)
        ⑤ 분석 목적이 있는가?        없으면 기본값: 내부 검토용

STEP 2. 시나리오 고지
        → "시나리오 [A/B/C] 기준으로 진행하겠습니다. 맞을까요?" 한 번 확인

STEP 3. 기술 분류 제안
        → 기술명 기반으로 REPORT_TEMPLATE.md의 분류 기준에 따라
          기술 분류 체계(A~D) 초안 제안 → 사용자 확인 후 확정

STEP 4. 데이터 처리
        → 첨부 파일 있으면 핵심 수치 추출
        → 없으면 Claude 지식 기반 추정치 준비

STEP 5. 보고서 생성
        → REPORT_TEMPLATE.md 구조에 따라 9챕터 순서대로 작성
        → 시나리오별 Ⅶ·Ⅷ·Ⅸ장 관점 적용

STEP 6. 차트 생성 (Python matplotlib)
        → 기본 4개 + 시장 환경 시각화 2개 = 총 6개 차트
        → NanumGothic 폰트 직접 경로 지정 필수 (한글 깨짐 방지)

STEP 7. Word 파일 생성 (Node.js docx)
        → 차트 이미지 삽입 위치는 REPORT_TEMPLATE.md 기준

STEP 8. 검토 요청
        → 생성 완료 후 "이 방향이 적합한지, 수정이 필요한 부분이 있는지" 확인 요청
```

## 3. 첨부 파일 유형별 처리 방법

### 특허 데이터 (CSV / Excel)
```
기대 컬럼: 출원번호, 출원일, 출원인, 발명의명칭, IPC코드, 국가
처리:
  - 연도별 출원 건수 집계 → Chart 4 데이터
  - 출원인별 건수 집계  → Ⅲ장 주요 출원인 테이블
  - IPC 코드 빈도 집계  → Ⅲ장 IPC 동향
  - 컬럼명 불일치 시    → 내용 보고 자동 매핑
```

### 논문 데이터 (Excel / CSV)
```
기대 컬럼: 제목, 저자, 발표연도, 저널/학회, 키워드, 초록
처리:
  - 연도별 논문 건수 집계 → Chart 1 데이터
  - 키워드 빈도 분석      → Chart 2 데이터
  - 주요 저자/기관 집계   → Ⅴ장 연구 그룹 테이블
  - 초록 분석             → R&D 흐름 서술에 활용
```

### 보유 특허 명세서 PDF → 확장 모듈 A 자동 실행
```
  1. 특허번호, 출원인, 출원일 확인
  2. 독립항(청구항 1번) 전문 추출
  3. 핵심 구성요소 3~5개 식별 및 요약
  4. Ⅷ장 공백기술과의 연결 가능성 분석
  5. "보유IP × 공백기술 연결 지도" 테이블 생성
  → 시나리오 B에서만 의미 있는 분석임을 사용자에게 안내
```

### 경쟁사 특허 명세서 PDF → 확장 모듈 B 자동 실행
```
  1. 특허번호, 출원인, 법적 상태 확인
  2. 독립항 전문 추출 및 구성요소 분해
  3. 의뢰 기관 연구와의 저촉 여부 예비 검토
  4. Design-around 방향 3개 이상 제안
  5. FTO 예비 검토 리포트 생성
  6. 반드시 "전문 변리사 검토 필요" 문구 포함
```

## 4. 콘텐츠 작성 품질 기준

### 깊이 기준
- **전문가 수준**: 기술 원리, 설계 방정식, 트레이드오프 관계를 구체적으로 서술
- **표면 요약 금지**: "A 기술은 중요하다"처럼 내용 없는 서술 사용 금지
- **수치 근거 포함**: 효율, 건수, 연도 등 구체적 수치 반드시 포함

### 객관성 기준
- 추정치와 사실을 명확히 구분
- 시장 수치는 출처 병기 또는 `(E)` 표시. **단일 수치보다 범위(range)로 제시**
- 특허번호·논문 인용은 확인된 정보만 사용
- **의뢰 기관 정보가 없는데 특정 기업 역량과 연결 짓는 서술 금지**

### 전략성 기준
- 분석으로 끝내지 않고 시나리오에 맞는 행동 방향을 도출
- 공백기술은 실현 가능성·시급성을 함께 제시
- Ⅸ장 과제표는 즉시/단기/중기/장기로 반드시 구분

### 한글 폰트 코드 패턴 (반드시 준수)

**폰트 경로는 단일 값으로 고정하지 않고, 후보 목록 중 실제 존재하는 파일을 탐색하여 사용한다.**
환경마다 설치된 한글 폰트가 다를 수 있음(예: NanumGothic이 없고 Noto Sans CJK KR만 있는 환경 실사례 있음).

```python
import os
import matplotlib.font_manager as fm
import matplotlib.pyplot as plt

FONT_CANDIDATES = [
    '/usr/share/fonts/truetype/nanum/NanumGothic.ttf',
    '/usr/share/fonts/opentype/noto/NotoSansCJK-Regular.ttc',   # Noto Sans CJK (KR 서브셋 포함)
    '/usr/share/fonts/truetype/noto/NotoSansCJK-Regular.ttc',
]
FONT_PATH = next((p for p in FONT_CANDIDATES if os.path.exists(p)), None)
if FONT_PATH is None:
    raise RuntimeError("사용 가능한 한글 폰트를 찾지 못함 — fc-list로 설치된 CJK 폰트 확인 필요")

fm.fontManager.addfont(FONT_PATH)
prop = fm.FontProperties(fname=FONT_PATH)
plt.rcParams['font.family'] = prop.get_name()
plt.rcParams['axes.unicode_minus'] = False

fp = fm.FontProperties(fname=FONT_PATH, size=11)
ax.set_title('제목', fontproperties=fp)
ax.set_xlabel('축 레이블', fontproperties=fp)
ax.legend(prop=fp)
# 모든 텍스트 요소에 fontproperties 직접 지정 — rcParams 단독 사용 금지
```

실제 사용한 폰트가 NanumGothic이 아닌 대체 폰트인 경우, 보고서 하단이나 검토 요청 시 어떤 폰트로 대체되었는지 사용자에게 고지한다.

## 6. Word 파일 생성 기준

```
폰트: Malgun Gothic (본문), Arial (숫자/영문)
페이지: A4 (11906 × 16838 DXA)
여백: 상하 1134, 좌 1417, 우 1134 (DXA)
헤더: 보고서명 | 기술동향 분석 보고서
푸터: 보고서명 | 내부검토용 | 페이지번호/전체페이지
      → 의뢰 기관명이 있는 경우에만 푸터에 기관명 추가
테이블 헤더 배경: #1F4E79 (흰색 텍스트)
테이블 짝수행 배경: #EBF3FA
강조 컬러: PRIMARY #1F4E79, SECONDARY #2E75B6
ShadingType: 반드시 CLEAR (SOLID 사용 시 배경 검게 됨)
테이블 width: 반드시 DXA 단위 (PERCENTAGE 금지)
테이블: columnWidths 배열 합계 = 테이블 전체 width와 일치해야 함

푸터 전체 페이지 수 필드: docx(npm) 라이브러리에는 `NumberOfPages`라는 export가 존재하지 않음(실사용 중 확인된 오류).
전체 페이지 수는 `PageNumber.TOTAL_PAGES`를 `TextRun`의 `children` 배열에 넣어 사용한다.
예: `new TextRun({ children: [PageNumber.CURRENT] })`, `new TextRun({ children: [PageNumber.TOTAL_PAGES] })`
```

## 7. 사용자 응답 원칙

- 보고서 생성 전: 시나리오 확인 + 기술 분류 제안을 합쳐서 **한 번만** 확인 요청
- 챕터별 중간 보고 없이 전체 생성 후 한 번에 검토 요청
- 수정 요청 시: 해당 챕터만 부분 수정 (전체 재생성 금지)
- 결과 전달 후 반드시 "이 방향이 적합한지, 수정이 필요한 부분이 있으신지 검토 부탁드립니다" 형태로 마무리
